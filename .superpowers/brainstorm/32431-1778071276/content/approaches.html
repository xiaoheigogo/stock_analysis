三种实现方案对比

基于数据库探查结果：13.2M 行 / 4559 只股票 / 清洗率仅 1.5%

A

严格按 Phase 1 Plan 原样顺序执行

Step 1→2→3→4→5 依次推进

优点

  • 与文档一致，可追溯
  • 每步有明确验收标准

缺点

  • 部分步骤可并行却串行了
  • trade_date 更新不阻塞后续步骤
  • 预估 5 天，实际可能更长
B

管线优先 - 跳过非阻塞步骤直奔核心

Bug修复 → 清洗 → 训练 → 预测，数据补齐后置

优点

  • 最快看到预测结果
  • 先验证核心链路可行性
  • 约 2-3 天完成

缺点

  • 数据 quality 风险累积
  • 技术债后移
  • 可能遗漏边界 case
C

混合分阶段并行 ⭐ 推荐

三阶段：准备→核心→集成，每阶段内部可并行

优点

  • 修复关键 bug (SQL注入) 优先
  • 数据补齐与清洗可后台进行
  • 训练可 4 模型串行但整体进度快
  • 干净基础进入 Phase 2

缺点

  • 需要协调三个阶段
  • 预计 3-4 天

方案 C 详细分解

三阶段工作流

🔧 阶段一：准备

  • 修复 SQL 注入 (is_cleaned)
  • 修复硬编码日期
  • 更新 trade_date 表
  • 清理冗余文件
  • 增量补齐数据缺口

预计: 0.5 天

🧠 阶段二：核心

  • 清洗 3-5 只精选股票
  • 串行训练 4 模型 (LSTM→GRU→CNN-LSTM→Transformer)
  • 对比指标选出 best_model
  • 保存 best_model.pth

预计: 2 天

🚀 阶段三：集成

  • 模型接入 recommendation_system
  • 实现 main.py predict 命令
  • 更新 FastAPI 端点
  • 前端替换模拟数据
  • 新增预测列表/详情页

预计: 1 天
